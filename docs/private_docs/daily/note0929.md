1.优先级/异构性：job本身和小车各自之间也有不同

2.lifelong / MAPD / JSSP 任务流；

因为是life long的MAPF场景,所以小车一定是一直运行的,唯一的不同就是job本身的复杂度。

我现在的思路是：

1.简化场景：对于搬运类任务job（只执行一次搬运operation+送回即可），指定job本身的优先级，
分配给同质的小车，进行划分，小车继承job优先级（因为只有一次所以operation的优先级=job的优先级。
查看划分了优先级的job完成任务的快慢。 （此处有问题， 如何在分级前提下对job进行评价？小车继承job优先级后，
如何利用优先级进行寻路判定？）

2.复杂场景：对于复杂的job（执行多次operation），计算operation的优先级，不断分配给同质的小车,
小车继承operation的优先级,进行MAPF寻路,查看job相关完成的指标。（此处有问题，如何计算operation的
优先级，根据什么计算优先级，因为这里分级的目标是发掘job中的**关键节点**，所以可能已有的静态方案会给我们答案。）

能不能进行生成式的寻路？之前只是读取当前

我最终能否将这两个场景用统一的端到端模型完成？输入job列表,直接输出小车运输的operation方案即可。
我当前有点没想明白，为啥需要小车自动寻路？如果我operation静态分配给固定的机器会怎么样？我感觉必然
会浪费其他的机器的算力，然后就是有些机器负载太高，整体吞吐量也会下降，同时过于集中的话，小车之间的
碰撞也会更多，车密度太大确实也不太方便。那么其实一个启发性的思路就是尽量让全局的机器负载均衡，
但是！如果仅仅如此的话，路由的时间会被拉长，因此如何充分利用机器，不产生拥堵又不造成过大的路由开销
就是我们需要考虑的问题。

（计算job优先级，计算operation优先级，分配operation给小车运输，）
