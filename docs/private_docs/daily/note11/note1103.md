


实际上确实可以分为两类，

1. 静态的分配：job → operation → 进行机器选择 → routing_task → agv_assignment
这个流程模拟了传统的JSSP问题，但是缺乏实时性，调整代价会比较高。
这类 routing task 其实是一个“确定性搬运任务”，可以看作一个完整的运输指令。
此时routing task可以包含的信息为：
| 字段               | 含义                             | 示例          |
| ---------------- | ------------------------------ | ----------- |
| `task_id`        | 唯一任务标识                         | `"T001"`    |
| `job_id`         | 任务所属的 Job                      | `"J05"`     |
| `op_id`          | 对应的 Operation                  | `"OP2"`     |
| `agv_id`         | 执行的 AGV（可后填）                   | `"AGV-3"`   |
| `source`         | 起点坐标（当前工位 / 缓存区）               | `(12, 5)`   |
| `destination`    | **目标机器的输入点坐标（固定）**             | `(22, 10)`  |
| `machine_id`     | 已分配的机器 ID                      | `"M12"`     |
| `material_type`  | 搬运物料类型（如工件 A/B）                | `"A"`       |
| `status`         | `pending / in_progress / done` | `"pending"` |
| `priority`       | 调度优先级（如紧急任务）                   | `1`         |
| `estimated_time` | 预估耗时（路径长度/速度）                  | `23.5`      |



2. 动态的分配：job → operation  → routing_task → 进行机器选择 → agv_assignment
AGV 与 Machine 的分配解耦，更贴近实际智能工厂流程。
这时的 routing task 更像一个 “待决策任务 (pending decision task)”，它代表“有工件要运去可加工机器区”，但具体是哪台机器要根据现场状态实时决定。
此时routing task可以包含的信息为：
| 字段                   | 含义                                               | 示例                      |
| -------------------- | ------------------------------------------------ | ----------------------- |
| `task_id`            | 唯一任务标识                                           | `"T002"`                |
| `job_id`             | 所属 Job                                           | `"J07"`                 |
| `op_id`              | 对应 Operation                                     | `"OP1"`                 |
| `candidate_machines` | **候选机器列表**                                       | `["M01", "M03", "M04"]` |
| `machine_area`       | 可加工机器区域（矩形/多点集合）                                 | `[(20,10), (23,12)]`    |
| `source`             | 起点坐标（当前缓存区）                                      | `(12, 5)`               |
| `destination`        | `None` 或 `Dynamic`（在执行中决定）                       | `None`                  |
| `decision_rule`      | 动态决策函数引用，如 `select_least_busy_machine`           | callable                |
| `status`             | `pending / traveling / waiting / decided / done` | `"traveling"`           |
| `context`            | 运行时上下文（如环境状态 snapshot）                           | `{load: 0.8}`           |



| 维度                | 静态分配            | 动态分配               |
| ----------------- | --------------- | ------------------ |
| RoutingTask 的生成时机 | 在 Job 分解阶段      | 在 Operation 就绪阶段   |
| 是否包含 destination  | ✅ 是             | ❌ 否（延迟决策）          |
| 调度决策位置            | 调度器全局统一决策       | 环境或智能体局部决策         |
| 路径计算              | 可预先规划           | 需实时调整              |
| 适用场景              | 批量调度、静态车间       | 智能制造、实时柔性车间        |
| 算法适配              | OR-Tools / MILP | MARL / PPO / 自适应调度 |


解析 Job 集合的目标是将逻辑任务（Job/Operation）映射为物理执行任务（Routing + Scheduling + Assignment）。
换句话说，你要从“计划层”数据生成“调度层”可用的数据结构。

JobSolverResult 的定位：静态调度器的产物

它的作用是描述一次完整的调度解（schedule），即：
所有机器的作业顺序；
每个操作的执行时间区间；
每个操作对应的传输请求；
统计指标（makespan、延迟、负载等）。

JobSolverResult = 计划层（planning layer）的输出
此时的 result.machine_schedule 就是所有机器的任务甘特图。
transfer_requests 可以直接生成 AGV 路由任务（RoutingTask）。

RoutingTask 的定位：执行层（execution layer）的任务粒度
RoutingTask 更像是“车队控制层”的输入单元。
它的关注点是单次搬运任务，例如：
把某个 operation 的输入物料从 source 送到 destination；
或者在动态场景中，仅确定 source，destination 由实时调度模块选择。
它强调“任务的可执行性（where to go）”，
而非“全局的时间调度（when to do）”。

在动态环境中，JobSolverResult 通常只保留结构上的辅助功能，比如：

@dataclass
class JobSolverResult:
    op_meta: Dict[Tuple[int, int], Dict]  # 描述操作属性，如加工时间、资源需求
    stats: Dict  # 动态执行过程中统计信息
**而 machine_schedule 和 transfer_requests 不再预生成**
而是：每次根据系统状态（机器空闲、AGV位置、物料到达等）即时生成；
由调度器/决策器（policy）输出新的 RoutingTask；
动态地拼出新的 operation 执行序列。
