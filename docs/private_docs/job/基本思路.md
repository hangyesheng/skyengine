多个可选求解器（伪代码 + Python 可运行实现）：priority_greedy、milp_compact（若安装 pulp/ortools 可用）、local_search（基于交换/插入的改进）

输出格式说明（machine schedule + op-level metadata + transfer 请求清单）—— AGV 路由器只需消费 transfer 请求并返回实际运输完成时间或运输计划

推荐的 job 专用指标 & 绘图示例（machine-centric 甘特图、负载平衡图、等待时间直方图）



Job 求解器只负责将 operations 分配到 machines、给出顺序与开始时间（基于 假设的 运输/传送延迟占位或 0 延迟），并标注哪些 consecutive ops 需要跨机转移（由 AGV 模块负责调度运输并返回实际 transfer 完成时间，供后续再调优/仿真）。


## 设计要点

Solver 不分配具体哪辆 AGV，只产生 transfer_requests：包含 ready_time（该 op 的前序完成时间）和预计接收机器（assigned_machine）。AGV 模块负责返回实际 transfer_complete_time（或具体 route），然后上层可以把该实际 transfer 完成时间反馈给 Solver/Simulator 用于真实开始时间修正／仿真推进。

允许传入 transfer_time_estimator(machine_from, machine_to) callback（如果你想让 job-level solver 在评估 start_time 时用 AGV 给出的估计值，可以传入一个估计器函数）。如果你不想让 job-solver考虑 AGV，传入恒为 0 的估计器。

支持多机器候选（machine_options），约束例如 machine 容量（单位机器单工作序列）和 precedence（job 内部顺序）。



## 设计要点

### priority_greedy（最简单、速度快）

把所有可执行的 operation 用优先级队列（如 FIFO / SPT / EDD / 自定义权重）排序。

对每个 op 选取使得早完成时间最小的 machine（考虑 machine 空闲时间 + 可选 transfer_time_estimator）。

输出 sequence 与时间。

适合大规模近实时 baseline。

### milp_compact（中小规模求精确解）

紧凑模型：为每 op 选 machine（离散），为每 op 决定开始时间（整数或连续），加 precedence 与机台不重叠约束（采用 disjunctive pairwise 或二进制顺序变量）。

对于中等规模 (~几十 ops) 可用；对大规模会爆炸。

可用 pulp 或 OR-Tools CP-SAT。

### local_search（启发式改进）

先用 greedy 得到初始解，再做 pairwise swap / reinsert / relocate（在同机或跨机但符合 candidate）尝试降低 makespan 或 tardiness。

适用于希望在给定时间内改进初解的场景（可在 RL 训练或仿真之间作为子过程调优）。

输出与 AGV 对接约定（非常重要）
{
  "job_id": int,
  "op_id": int,
  "from_machine": int,      # None if same machine or first op
  "to_machine": int,
  "ready_time": float       # earliest ready to be transported (prev op finish)
}
AGV 模块返回的每个请求的响应可以是：
{
  "job_id": int,
  "op_id": int,
  "transfer_complete_time": float,
  "agv_id": int,            # optional
  "route": [...]            # optional details
}

上层仿真/调度器可以把 transfer_complete_time 与 op 的 est_start 相结合，得到真实 start_time = max(machine_available_time, transfer_complete_time, release_time)。


上层 RL（或调度器）发出 job 策略 → job-solver 分配 machine/sequence → 生成 transfer_requests → AGV 模块调度 transport → 把 transfer 完成时间反馈 → 仿真推进 / 生成 reward。



推荐两种集成模式：

批次联动（offline-ish）：job-solver 先给出完整 schedule（用估计 transfer_time 或 0），AGV 模块按请求计划并返回实际 transfer times，仿真用实际 times 重演并评估（适合评估比对）。

在线联动（event-driven）：每次 op 完成产生 transfer_request，AGV 调度后返回 transfer_complete -> job-level 调度器或仿真引擎根据实际 transfer 完成触发后续 op 开始（更真实，适合 RL 在交互中学习）。我建议你如果已有 AGV 路由模块，优先使用“事件驱动”集成。
