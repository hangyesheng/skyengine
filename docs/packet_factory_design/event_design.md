
# 🚩 事件系统设计文档

---

一、设计背景与目标

在复杂的系统模拟与调度环境中，事件（Event）是一种重要的动态机制。为了实现系统状态的动态变化、故障模拟、性能扰动等功能，需要设计一个具备高解耦性、高扩展性、易用性的事件管理系统。

✅ 系统目标

- 每个事件类自带处理逻辑（trigger 和 recover 等方法）。

- 事件支持通过配置文件动态加载，无需硬编码。

- 具备良好的扩展性，支持无侵入式地添加新事件。

- 提供事件队列管理机制，根据时间驱动事件触发。

- 支持事件的完整生命周期管理，包括触发、恢复和追溯。

按照这种方式感觉组件不需要修改执行流程，进行状态转换即可

# 二、核心组件设计

## 1️⃣ Event（事件）
✔️ 基础功能

- 每个事件是一个独立的类。

自带核心方法：

- trigger(env)：触发事件，告诉某个组件 如何修改自身状态。

- recover(env)：恢复事件，撤销影响。

- （可扩展）pre_check(env)：触发前检查。

- （可扩展）post_process(env)：触发后收尾。

也就是说组件本身对事件是也是有感知的。因此组件内部含有check_event函数，判断
上个时间片自己身上发生了什么事件。

## ✔️ 状态管理
事件包含内部状态：

- 事件ID

- 开始时间、恢复时间

- 当前状态（待触发、已触发、已恢复）

## 2️⃣ EventManager（事件管理器）
✔️ 功能

- 管理所有已注册的事件类型。 

- 根据配置文件动态创建事件实例。 

- 提供事件的工厂方法。

✔️ 特点

- 不直接参与事件的调度，仅负责事件的声明与实例化。

- 新增事件使用前需声明注册，无需修改其他代码。

## 3️⃣ EventQueue（事件队列）
✔️ 功能

- 管理当前激活的事件列表。

- 按照时间驱动，检测并触发符合条件的事件。

- 管理事件的恢复逻辑。

✔️ 运行机制

- 每个 env_step 结束时，调用 event_queue.step(current_time)。

- 弹出小于等于当前时间的所有待触发事件，依次执行 trigger()。

- 检查需要恢复的事件，执行 recover()。

# 三、总结

该事件系统设计具备以下优势：

🏗️ 模块化：事件、事件管理器、事件队列分离。

🔧 配置驱动：无需更改代码即可调整事件行为。

♻️ 生命周期完整：支持事件触发、恢复与历史追溯。

🚀 易于扩展：新增事件零侵入。
